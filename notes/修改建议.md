## 第一轮审阅 — 2026-04-17

### 一、全局性问题（Overall Issues）

#### 1. 实验数据完全缺失（最严重的问题）
- **位置**: `chap_experiments.tex` Table 7.1, Table 7.2
- **问题**: 所有结果表格都是 `---`，没有任何实际数据。整个 Chapter 7 读起来像是一个"实验计划"而非"实验结果"。这在一篇 4YP 论文中是不可接受的。
- **建议**: 
  - 必须跑完实验并填入真实数据
  - 如果时间不够跑完所有配置，至少需要 M0 和 M1 的完整数据，M2/M3 可以是部分数据
  - 如果某些实验确实无法完成，需要在 Limitations 里明确说明，而不是用 `---` 留空

#### 2. 过多 Bullet Point / itemize，不像学术论文
- **问题**: 几乎每个 section 都大量使用 `\begin{itemize}` 和 `\begin{enumerate}`。尤其严重的章节：
  - `chap_literature_review.tex`: 几乎全部是 bullet point 列举
  - `chap_architecture.tex` (Benchmark Design): Section 4.1 的工具列表、Section 4.4 的配置列表
  - `chap_failure_cases.tex`: Baseline Results 全是 bullet point
  - `chap_solution_proposal.tex`: MCC Specification, PDP 三个机制
  - `appendix_technical.tex`: 整个附录几乎全是 bullet list
  - `appendix_related_works.tex`: 每个系统分析都是 bullet point 模板
- **建议**:
  - Bullet point 只用于真正的列表项（如工具名称列表、对比表格）
  - 叙述性内容应该写成完整的段落（prose）。例如 Baseline Results 应该是分析性的段落，而不是三个 bullet point
  - 对比性内容考虑用表格替代 bullet list（已有的 Table 2.1 做得很好）
  - 特别是 Related Work 的 Appendix B，每个系统都是 `Core Contribution / Architecture / Limitations / Pulse's Advancement` 的模板化 bullet，读起来非常机械

#### 3. 语气偏向产品宣传而非学术论文
- **位置**: 多处
- **问题**: 部分语言更像创业 pitch 而非学术写作：
  - "No existing system occupies all six columns" (chap_literature_review, L115) — 过于 promotional
  - "Single agents are useful; networks of agents are transformative" (L57) — 没有引用支撑的断言
  - "Pulse as a 'Dropbox for Agents'" (appendix_related_works, L234) — 产品比喻，不适合学术论文
  - "The coordination overhead that consumes knowledge workers' time will not be solved by more capable individual agents" (conclusion, L75) — 过于绝对的断言
- **建议**: 用更谨慎的学术语言。"We observe that..." "Our results suggest..." "To the best of our knowledge, no existing system..."

#### 4. 自引用 vs 第三方引用比例失衡
- **问题**: 论文大量引用自己的系统设计（Pulse），但对比的第三方实验数据为零。没有引用任何其他 benchmark 的具体数值来做对比。
- **建议**: 
  - 在 Related Work 或 Experiments 中引用 TensorTrust、HackAPrompt 等的具体实验数据
  - 如果可能，在 TensorTrust 或其他公开 benchmark 上跑 Pulse 做对比

### 一、论文描述 vs 实际实现的严重脱节（最关键的新发现）

这一轮发现了 `experiment.md` 中明确记录的 "thesis-to-code gap"。这些不是小问题，是论文诚信层面的：

#### 1. MCC 的描述与实现不匹配
- **论文说**: "entire execution environment constructed de novo" (chap_solution_proposal.tex L69)，类比 Docker/SELinux 级别的进程隔离
- **实际实现** (来自 experiment.md L283): "tool filtering at route level" — 只是 API 层的参数校验
- **影响**: 论文的核心安全论点（"architectural guarantee analogous to process isolation in operating systems"）站不住脚
- **建议**: 
  - 选项 A: 实现真正的沙箱（太耗时）
  - **选项 B（推荐）**: 修改论文措辞，诚实描述实现级别。把 "Docker/SELinux" 的类比降级为 "analogous to API-level capability filtering, inspired by capability-based security principles"。保留 OS 类比作为设计灵感，但明确说明当前实现是 application-layer enforcement，不是 OS-level isolation

#### 2. Memory 系统的描述与实现不匹配
- **论文说**: "vector database with embedding-based retrieval and metadata filtering" (chap_solution_proposal.tex L35, appendix_technical.tex L189-208)
- **实际实现** (来自 experiment.md L284): "note folders with SQL ILIKE" — 只是文件夹 + 关键字搜索
- **影响**: Dual-Track Memory 的核心价值主张（"embedding-based retrieval with relationship_id filtering"）与实际系统不符
- **建议**: 修改论文，区分"设计"和"当前实现"。可以说 "We implement a simplified version using folder-based isolation with keyword retrieval; a full vector database implementation is future work"

#### 3. Macro Constitution 的描述与实现不匹配
- **论文说**: 结构化 JSON 规则 + precedent learning (chap_solution_proposal.tex L142-157)
- **实际实现** (来自 experiment.md L285): "freeform text in MEMORY.md"
- **建议**: 同上，区分设计与实现

#### 4. 这个 gap 的处理方式建议
- 在 Chapter 6 (Solution Proposal) 的每个方案后加一个 "Implementation Status" 小节，明确区分 design (what we propose) 和 prototype (what we built)
- 在 Limitations 里专门加一段关于 "Design-implementation gap" 的讨论
- 4YP 论文这样做是完全可以接受的——提出设计 + 部分实现 + 说明差距是诚实且有价值的贡献

## 第三轮审阅 — 2026-04-17（"人味"审阅：AI 生成痕迹、考官致命问题、论证断裂）

> 本轮核心问题：如果考官拿到这篇论文，哪些地方会立刻让他皱眉？哪些地方读起来"不像人写的"？

### 一、AI 生成痕迹（读起来不像人写的地方）

#### 1. 全文结构过于"模板化"——每章的 pattern 一模一样
整篇论文几乎每个 section 都遵循同一个模式：
- **Bold heading** 引入概念
- 一段概述
- `\begin{itemize}` 列举 3-5 个点
- **"The key insight/critical limitation/implication for our work"** 的总结句

具体例子：
- `chap_literature_review.tex`: 每个 subsection 结尾都有一个 `\textbf{The xxx for our work}` 总结句（L17, L25, L35 各一个）。人类写作者不会这么机械地给每段加相同格式的"takeaway"
- `appendix_related_works.tex`: 每个系统分析都是 **Core Contribution → Architecture → Limitations → Pulse's Advancement** 四段式。连续读 8 个系统都是这个格式，极度机械
- `chap_failure_cases.tex`: 5 个 attack class 的结构完全一样：Example → Observation → 总结。可以考虑把前两三个写详细些（narrative + 具体 trace），后两个写简短些，体现人类写作的自然详略取舍

**建议**: 
- 打破模板。让不同 section 有不同的写作节奏
- Related Work 里重要的工作（MemGPT, ReAct, MCP）写成完整的分析段落；次要的（ChatDev, LangGraph）可以在一两句话里带过
- 去掉千篇一律的 `\textbf{The xxx for our work}` 句式，换成自然的过渡句

#### 2. "三段式"论证的过度使用
论文有一种"三件事"的 AI 写作习惯。几乎所有重要论点都被拆成恰好三个子点：
- 三个贡献（contributions）— OK，这个合理
- 三个 trust tiers — OK
- 三个 memory tiers (L1/L2/L3) — OK
- 三个 PDP enforcement 机制 — 可以
- 三个 MCC limitation that baseline has — 可以
- 三个 gap in Chapter 5 summary — 到这里开始显得刻意
- 三个 sections in Related Work — 刻意
- 三个 bodies of literature (L4, chap_literature_review) — 刻意

**建议**: 审视每个"三"，如果某些地方真的有 2 个点或 4 个点，不要硬凑成 3 个

#### 3. 过于整洁的叙事弧——没有"挣扎"的痕迹
好的论文会留下探索过程中的"挣扎"痕迹：尝试了 X，发现不行，所以改成 Y。这篇论文读起来太"干净"了：
- 每个问题都恰好有一个对应的解决方案
- 5 个 failure class 恰好对应 3 个 architectural solution
- 没有提到任何"我们尝试过但失败了"的方案

从 `experiment.md` 可以知道实际上有很多挣扎（over-escalation 问题、sanitizer 的 threshold tuning、M3 utility 只有 33% 的问题），但论文里完全没有反映这些。

**建议**: 
- 在 Chapter 5 或 Chapter 7 加入 "Design Iteration" 或 "Lessons from Initial Experiments" 小节
- 提到 M3 的 over-escalation 问题（March 10 run: 33% utility）以及你们如何思考解决它
- 这些"失败"反而能增强论文的可信度——考官想看到你真的做了实验并从中学到了东西

#### 4. 论文的"声音"不一致
有些段落是 formal academic tone（Chapter 3 的数学形式化），有些像 blog post（Chapter 2 的 Metcalfe's law 段落），有些像产品文档（Appendix A 的 code snippets）。一篇论文需要统一的"声音"。

**建议**: 通读一遍，以 Chapter 3 的 tone 为标准，调整其他章节的语气

### 二、考官会问的致命问题

#### 5. "你自己写的 benchmark 测试你自己的方案，怎么证明不是 overfitting？"
这是最可能被考官质疑的问题。论文目前**完全没有**讨论这个 validity threat：
- Benchmark 是你设计的
- Evaluation queries 是你写的
- Solutions 是你提出的
- 人为因素可能导致 benchmark 刚好测试你的 solution 擅长的 scenarios

`reference.md` 里详细分析了 Yao 如何通过 ablation + human ceiling + sim-to-real validation 来回答这个问题。但论文本身没有借鉴这些。

**建议**:
- **必须**加一个 "Threats to Validity" 小节（放在 Chapter 7 或 Chapter 8 的 Limitations 里）
- 明确讨论 construct validity（benchmark 是否测了我们想测的东西？）和 internal validity（是不是 solution 恰好 overfit 了 benchmark？）
- 缓解措施：(a) human ceiling 作为独立参照，(b) cross-model testing 减少 model-specific overfitting，(c) 公开 benchmark 让他人复现

#### 6. "你的 threat model 为什么不考虑 colluding entities？"
Threat model (Chapter 3, L49-65) 假设每个 external entity 独立行动。但如果 E_i 和 E_j 串通呢？他们可以分别向 agent 提问，然后合并答案做 multi-query inference。

论文完全没有讨论 collusion attack。

**建议**: 在 Threat Model 的 scope boundary 说明中加一句 "We do not consider colluding entities in this work; we discuss this as future work in Section 8.4"

#### 7. "Heartbeat mode 是你的 novelty claim 之一，但实验中用了吗？"
- Abstract 和 Chapter 4 都强调 heartbeat mode 是一个 novelty
- Chapter 7 有一个 "Heartbeat Mode Results" section，但里面只有问句（"Does the heartbeat engine..."），没有任何回答
- 考官会问："你声称 heartbeat mode 是贡献之一，但你有没有实际评估它？"

**建议**:
- 要么填入 heartbeat 评估数据
- 要么降低 heartbeat 在 contributions 中的权重，把它移到 "design contribution / future work" 而非 "empirically validated contribution"

#### 8. "Dual-Track Memory 对 utility 的影响是什么？"
论文只讨论了 Dual-Track Memory 对 security 的好处（防止 cross-relationship leakage），但**从未量化它对 utility 的成本**。
- 隔离 memory 意味着 agent 在和 E_i 交互时无法利用从 E_j 学到的 common knowledge
- 例如：如果 Investor A 和 Investor B 都问同一个问题，agent 需要重新从 notes 检索，而不能利用之前已经 formulated 的回答
- 这是一个 utility cost，但论文假设它是零

**建议**: 在 Chapter 6 Section 6.1 或 Chapter 7 中讨论这个 trade-off

### 三、论证链断裂处

#### 9. Chapter 2 → Chapter 3 的过渡缺失
Chapter 2 结尾（L131-132）讨论的是 "Personal Agent Coordination Layer" 的定义。Chapter 3 开头直接跳到数学形式化。缺少一个过渡段落，说明"基于 Related Work 的分析，我们发现以下 gap 需要形式化..."

#### 10. Class 3 攻击（Multi-Query Inference）的防御方案缺失
- Chapter 5 明确说 Class 3 是"最难防御的"（L39）
- Chapter 6 的三个解决方案分别对应 Class 5 (Memory)、Class 1/2/4 (MCC)、和 continuous state (Escalation)
- **Class 3 没有对应的防御方案！** Chapter 6 没有任何一个 section 专门讨论如何防御 multi-query inference
- Conclusion 把 differential privacy 列为 future work（L64），但正文中连 acknowledgement 都没有
- 考官一定会注意到这个 gap

**建议**: 
- 在 Chapter 6 或 Chapter 7 中明确承认 "Class 3 remains an open challenge in our current architecture"
- 可以在 Escalation Protocol 中加一段讨论："The Sanitisation Agent may detect some inference patterns, but we do not claim reliable defence against this class"

- 论文假设每个 relationship 有一个 policy file（Chapter 4, L59-73）

- 如果是自动的，用什么逻辑？

**建议**: 在 Chapter 4 的 Policy File section 加一段说明 policy file 的创建机制。可以说 "In our current design, policy files are created by the owner based on trust tier templates; automated policy generation is future work"

#### 12. Metcalfe's Law 的引用缺失且论证薄弱
- `chap_literature_review.tex` L57: "an observation formalised by Metcalfe's law for computer networks"
- 没有引用 Metcalfe 或任何来源

**建议**: 要么加引用并说明这是一个 analogy（"Drawing an analogy to Metcalfe's law..."），要么直接移除这个 claim

### 四、文字层面的"不像人"之处（具体句子）

#### 13. 过度使用 em-dash (—)
全文大量使用 `---` (em-dash)。虽然 LaTeX 中 em-dash 很常见，但以下使用方式暴露了 AI 写作习惯：
- "agents cannot safely communicate across trust boundaries" — 整篇论文有几十处用 em-dash 做 parenthetical aside
- "no existing benchmark captures this dual measurement" — 用 em-dash 做总结
- 特别是 `chap_literature_review.tex` 几乎每段都有 1-2 个 em-dash

**建议**: 把一半的 em-dash 改成逗号、分号、或单独的句子。Em-dash 每页最多用 1-2 次

#### 14. "Crucially" / "critically" 的过度使用
- "Crucially, these solutions operate at the architectural level" (chap_failure_cases L112)
- "Most critically, it assumes agents have unrestricted access" (chap_literature_review L51)
- "And crucially, no existing benchmark measures both dimensions" (chap_introduction L28)

**建议**: 删除大部分 "crucially/critically"。如果论点真的 crucial，行文本身应该传达出来，不需要副词标注

#### 15. 每章都有 "这一章做什么" 的开头段
- Chapter 1: "This thesis tackles that question" (L10)
- Chapter 2: "This chapter reviews three bodies of literature" (L4)
- Chapter 3: "This chapter formalises the context-security trade-off" (L4)
- Chapter 4: "This chapter describes the benchmark we built" (L7)
- Chapter 5: "This chapter is diagnostic: we run the baseline configurations" (L4)
- Chapter 6: "This chapter proposes the treatment" (L4)
- Chapter 7: "This chapter evaluates the solutions" (L4)

这些"路标"单独看都合理，但 8 章连续每章第一句都是 "This chapter..." 的格式就显得机械了。

**建议**: 至少让一半的章节用不同的开头方式。例如 Chapter 5 可以直接从一个 motivating example 开始："Consider an agent with access to its owner's complete financial records..."

#### 16. Conclusion 和 Introduction 几乎是镜像
对比：
- Intro: "agents cannot safely communicate across trust boundaries"
- Conclusion: "how can AI agents coordinate across trust boundaries"
- Intro: "prompt-level defences have been shown to fail"
- Conclusion: "architectural enforcement—not prompt-level restriction—is the reliable path"
- Intro: "three contributions: benchmark, solutions, coordination layer"
- Conclusion: "three contributions: benchmark, solutions, coordination layer"

结论章几乎是引言的复述。好的结论应该**在更高的层面反思**，而不是重复 Introduction 的论点。

**建议**: 
- Conclusion 应该回答："在做完这个工作之后，我们对这个问题的理解有什么改变？"
- 加入反思性内容：什么让你惊讶？哪些假设被证伪了？如果重新开始你会做什么不同？
- 最后一段（"do not ask the agent to refrain..."）很好，这种反思性的总结多写一些
