\chapter{Introduction}
\label{chapter:introduction}

\section{The Rise of Personal Agents---and the Communication Gap}

The past two years have witnessed an explosion in AI agent capability. Systems such as ReAct \cite{yao2023react}, Toolformer \cite{schick2023toolformer}, and commercial products like Manus \cite{manus2025} and Devin \cite{devin2024} can now autonomously browse the web, write and debug code, manage calendars, and orchestrate multi-step workflows on behalf of a single user. These agents are rapidly evolving from reactive chatbots into persistent digital proxies with real-world authority.

Yet a critical gap remains: \textbf{agents cannot safely communicate across trust boundaries}. When a founder's agent needs to share a financial summary with an investor's agent, or when a researcher's agent must coordinate schedules with a collaborator's, there is no infrastructure for this interaction. The result is a ``coordination tax''---professionals still spend the majority of their time manually extracting context from AI assistants, reformulating it, and transmitting it through legacy channels such as email and Slack. The vast potential of agentic collaboration remains unrealised because agents are architecturally siloed.

This gap is not merely an engineering oversight---it reflects a missing layer in the emerging agent stack. Frameworks like LangChain \cite{langchain2023}, AutoGen \cite{wu2023autogen}, and CrewAI provide capabilities \textit{within} an agent; communication protocols like MCP \cite{anthropic2024mcp} standardise how agents connect to tools. But neither addresses the fundamental question: \textit{what should one agent be allowed to know about another agent's owner?} This thesis tackles that question.

\section{The Security Problem: Why Agents Remain Siloed}

The reason agents remain siloed is not a lack of ambition but a genuine security crisis. Granting an agent deep personal context---notes, emails, calendar, long-term memory---makes it maximally useful to its owner, but maximally dangerous when exposed to external parties. Prompt injection attacks can override system instructions to exfiltrate protected data \cite{greshake2023not}. Social engineering can manipulate an agent's reasoning. Confused deputy attacks can trick an agent into executing unauthorised actions. The attack surface is vast, and the stakes are personal.

Recent large-scale evaluations have confirmed the severity. Nasr et al.\ \cite{nasr2025attacker} tested twelve published prompt injection defences under adaptive attacks and found that \textit{every defence} could be bypassed with greater than 90\% success; human red-teaming achieved 100\%. Meta's ``Rule of Two'' framework \cite{meta2025ruletwo} formalises the danger: an agent becomes critically vulnerable the moment it simultaneously processes untrusted inputs, accesses sensitive data, and can take state-changing actions. Even state-of-the-art model-level hardening (Anthropic reports a 1\% attack success rate for Claude \cite{anthropic2025promptinjection}) leaves a nonzero residual risk that is unacceptable when personal data is at stake.

\section{The Security--Utility Paradox}

The research community has responded with a wave of defences---content classifiers, adversarial training, chain-of-thought filtering, and multi-layered sanitisation pipelines \cite{google2025promptinjection, anthropic2025promptinjection}. These approaches improve the average case but share a fundamental limitation: \textbf{they trade information exchange utility for security}. Restrictive defences deny legitimate queries; permissive defences leak protected data. Manual permission management reintroduces the very coordination overhead that agents were supposed to eliminate.

This creates a paradox at the heart of agentic communication:

\begin{quote}
\textit{Maximum coordination utility requires maximum context access, but maximum context access creates maximum security vulnerability when interacting across boundaries.}
\end{quote}

Current solutions cluster at the extremes of this trade-off. Prompt-based guards offer high utility but catastrophic security. Manual access control lists offer perfect security but crippling overhead. No existing system navigates the Pareto frontier between the two. And crucially, \textbf{no existing benchmark measures both dimensions simultaneously}---agent capability benchmarks ignore security, and agent security benchmarks ignore the utility cost of their defences. Without a benchmark that captures the full trade-off, progress on this problem cannot be measured.

\section{Research Approach and Contributions}

Our approach follows a \textit{benchmark-first} methodology: we first build the evaluation infrastructure that reveals where agents fail, then design solutions motivated by those specific failures. This structure---shared with recent agent research such as Yao's work on WebShop and ReAct \cite{yao2022webshop, yao2023react}---ensures that our solutions address empirically observed problems rather than hypothetical ones.

Concretely, the thesis makes contributions along two threads:

\begin{enumerate}
    \item \textbf{A benchmark for cross-boundary agent communication.} We design a benchmark in which a personal agent, equipped with realistic context (notes, calendar, email), interacts with external entities across varying trust tiers. We red-team these interactions to systematically catalogue failure modes---from direct prompt injection to multi-query inference attacks---and measure both the \textit{utility} of legitimate task completion and the \textit{security} of data protection simultaneously. We introduce a \textbf{heartbeat mode} in which agents operate autonomously at regular intervals, enabling evaluation of proactive (not just reactive) agent behaviour. No prior benchmark captures this dual measurement across a trust-tier gradient.

    \item \textbf{Architectural solutions motivated by benchmark failures.} Each failure class discovered in the benchmark motivates a specific architectural intervention:
    \begin{itemize}
        \item \textbf{Dynamic Sandboxing}: Capability-based execution environments (``Mountable Context Cells'') that physically remove unauthorised tools and data from the agent's context, providing architectural rather than prompt-level security \cite{dennis1966programming}.
        \item \textbf{Relationship-Native Memory}: A dual-track memory architecture that isolates episodic memory per relationship, preventing cross-contamination while maintaining perfect context recall within each relationship.
        \item \textbf{Adaptive Escalation}: An intelligent escalation protocol that learns social boundaries through precedent clustering, reducing manual permission overhead over time while maintaining zero-trust guarantees.
    \end{itemize}
\end{enumerate}

We address four research questions:

\begin{itemize}
    \item \textbf{RQ1}: What are the failure modes when AI agents communicate across trust boundaries?
    \item \textbf{RQ2}: How does a policy file (access rules) affect both utility and security?
    \item \textbf{RQ3}: How should agent memory be redesigned for multi-party interactions?
    \item \textbf{RQ4}: Can an adaptive escalation mechanism learn social boundaries and reduce manual overhead?
\end{itemize}

The remainder of this thesis is organised as follows. \Cref{chapter:related_work} reviews related work on agent capabilities, agentic security, and communication protocols. \Cref{chapter:problem_formulation} formally defines the context--security trade-off and states our research questions. \Cref{chapter:benchmark_design} presents the benchmark design, including our agent infrastructure and heartbeat evaluation mode. \Cref{chapter:failure_cases} reports the failure cases and insights from baseline experiments. \Cref{chapter:solution_proposal} proposes our three architectural solutions. \Cref{chapter:experiments} evaluates these solutions against the benchmark. \Cref{chapter:conclusion} concludes with a discussion of limitations and future directions.
